The strongest Wordle solvers end up looking less like a “neural network that predicts words” and more like a tiny search engine for information. The trick is realizing that Wordle isn’t about guessing the answer directly. It’s about maximizing how much uncertainty collapses each turn.

your current system does this:board state → neural net → guess

That works, but it throws away a lot of structure that Wordle gives you for free.

The architecture that consistently beats most ML-only solvers looks more like this:board state

    ↓

candidate answer set

    ↓

policy network proposes ~30 guesses

    ↓

exact entropy calculation

    ↓

choose best guess
Think of the neural network as a search accelerator, not the decision maker.
First ingredient: the candidate universe.

At every turn you maintain the set:possible_answers
You already do this with your filtering. If 2300 answers shrink to 12, the game is almost solved.

The amount of uncertainty left is literally:bits = log2(len(possible_answers))
Wordle starts around 11 bits of uncertainty.

Every guess should ideally remove 3–5 bits.
Second ingredient: true entropy scoring.
For a guess g, you simulate the result against every possible answer:
pattern = get_pattern(g, answer)
That creates buckets:
⬛⬛⬛⬛⬛ : 120 answers
⬛🟨⬛⬛⬛ : 42 answers
🟩⬛⬛🟨⬛ : 8 answers
...
Then compute:entropy = Σ -(p * log2 p)
This tells you how much information the guess is expected to reveal.

A perfect guess would split answers into even buckets, like a good binary search.
Third ingredient: policy network (the smart accelerator)
Computing entropy for all ~12,000 words every turn is expensive. So instead:
NN(board_state) → 30 promising guesses
Then you run exact entropy on those 30.

That gives you nearly the same quality as brute force, but much faster.

Your current model already acts like this — but the missing piece is forcing it to respect the candidate set.
Fourth ingredient: solution-aware guessing.

Sometimes the best entropy guess is not a valid solution.

Example:

Possible answers:BUNKY
GUNKY
HUNKY
JUNKY
Guessing MOUND might eliminate three at once.

But humans prefer guessing HUNKY because it might solve immediately.

Strong solvers blend both goals with a score like:score = entropy + λ * probability_of_being_solution
Where λ is small.
Fifth ingredient: endgame brute force.
When candidates drop below ~20, the solver should switch strategy:
for guess in candidates:
    simulate worst-case bucket size
choose guess minimizing worst-case
That becomes a minimax search, guaranteeing faster solves.
final architecture 1) Filter possible answers
2) If <20 → minimax search
3) Else:
      NN proposes guesses
      compute entropy exactly
      combine entropy + solve probability
      choose best
This hybrid solver typically solves Wordle in 3.4–3.6 guesses average, which is close to the theoretical optimum.
